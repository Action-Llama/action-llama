Running `al start` on your laptop works for development, but for production you want agents running 24/7 on remote infrastructure — no laptop required, automatic restarts, and reliable webhook delivery.

## When to deploy

You're ready to deploy when:

- Agents need to run continuously (not just during your work hours)
- Webhooks must be reachable from external services (GitHub, Sentry, etc.)
- You need TLS termination for secure webhook delivery
- You want automatic restarts if an agent crashes

## Deployment options

Action Llama supports two deployment approaches, both using VPS servers you control.

### SSH push deploy (`al push`)

The recommended approach. Manages your project on a remote server via SSH — no container registry, no cloud console, no vendor lock-in.

```bash
# One-time setup
al env init production --type server
al env set production

# Deploy
al push -E production
```

**How it works:**
- Project files are synced via rsync over SSH
- Docker images are built directly on the server (`tar | ssh docker build`)
- Credentials are pushed over SSH to the server filesystem
- The scheduler runs as a systemd service with automatic restarts
- Single-agent hot-reload: `al push dev -E production` updates one agent without restarting

**Best for:** Most deployments. Cost-effective ($5-6/month VPS), simple to operate, full control over the server.

See [VPS Deployment](/vps-deployment) for full setup, TLS configuration, and troubleshooting.

### Manual deployment

Install Action Llama directly on the server and run `al start` with the `--expose` flag. Simpler setup, but requires SSH'ing into the server to manage.

```bash
# On your VPS:
npm install -g @action-llama/action-llama
al new my-project && cd my-project
al doctor
al start -w --expose --headless
```

**Best for:** Quick setups, experimentation, or when you prefer to manage the server directly.

See [VPS Deployment — Manual Deployment](/vps-deployment#approach-2-manual-deployment) for systemd and TLS setup.

## Environment configuration

Both approaches use environment files to store server connection details. Environments live at `~/.action-llama/environments/<name>.toml` and are managed with `al env` commands.

```toml
# ~/.action-llama/environments/production.toml
[server]
host = "5.6.7.8"
user = "root"
keyPath = "~/.ssh/id_rsa"
basePath = "/opt/action-llama"
expose = true
```

| Command | Description |
|---------|-------------|
| `al env init <name> --type server` | Create a new environment file |
| `al env set <name>` | Bind the current project to an environment |
| `al env prov [name]` | Provision a new VPS (Vultr or Hetzner) |
| `al env check <name>` | Verify SSH connectivity and Docker availability |
| `al env list` | List all environments |

See [CLI Commands](/commands) for full `al env` reference.

## Server provisioning

You can bring your own server or let Action Llama provision one:

| Provider | vCPU | RAM | Storage | Price/month |
|----------|------|-----|---------|-------------|
| DigitalOcean | 1 | 1GB | 25GB SSD | $6 |
| Vultr | 1 | 1GB | 25GB SSD | $6 |
| Hetzner | 1 | 2GB | 20GB SSD | €4.15 |
| Linode | 1 | 1GB | 25GB SSD | $5 |

Automated provisioning (with cloud-init Docker install):

```bash
al env prov production       # Interactive: choose Vultr or Hetzner, pick region/plan
al env deprov production     # Tear down when done
```

Or connect to any existing server — any provider, any Linux box with Docker installed.
